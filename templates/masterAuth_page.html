Session-Based Authentication

Traditional server-side session management where user state is stored on the server.

Server Sessions Cookies Stateful
How it works:
  1. User logs in with credentials
  2. Server creates session and stores in memory/database
  3. Session ID sent to client via cookie
  4. Subsequent requests include session ID
  5. Server validates session on each request

OAuth with Google

Allow users to authenticate using their Google account via OAuth 2.0 protocol.

OAuth 2.0 Social Login Third-Party
Implementation Flow:
  1. User clicks "Login with Google"
  2. Redirect to Google OAuth consent screen
  3. User grants permission
  4. Google sends authorization code
  5. Server exchanges code for tokens
  6. Access user profile with tokens

JWT Authentication

Stateless authentication using JSON Web Tokens (Access & Refresh Tokens).

JWT Stateless Tokens
Token Flow:
  1. User logs in with credentials
  2. Server generates Access & Refresh tokens
  3. Tokens sent to client
  4. Access Token used for API requests
  5. Refresh Token used to get new Access Token
  6. Tokens verified via signature

Token Authentication Types

1. Bearer Tokens

Simple tokens included in Authorization header. Anyone with the token can access resources.

2. JWT (JSON Web Tokens)

Self-contained tokens with payload, signature, and optional encryption.

3. Refresh Tokens

Long-lived tokens used to obtain new access tokens when they expire.

4. One-Time Tokens

Temporary tokens for single-use actions like password reset.
